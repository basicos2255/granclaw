# FIX 124.1 - UI Status Binding to statusResolution

**Fecha**: 2026-05-05
**Estado**: Completado
**Autor**: Claude

## 1. Objetivo

Corregir la UI para que use `statusResolution.finalUiStatus` como fuente principal del estado visual, en lugar de inferir el estado de `decision.allowed`.

## 2. Problema

Aunque FIX 124 implementó el módulo `execution-status` y añadió `statusResolution` a las respuestas del backend, la pantalla `/control` seguía mostrando:

```
Usuario: "abrir calculadora"
UI mostraba: ✓ PERMITIDO
Realidad: statusResolution.finalUiStatus = 'setup_required'
```

### Causa raíz

En `Execute.tsx`, líneas 252-266, `resultStatus` se calculaba con lógica legacy:

```typescript
// Lógica anterior (incorrecta)
let resultStatus: ResultStatus
if (isMissingCapability) {
  resultStatus = 'missing_capability'
} else if (!allowed) {
  resultStatus = hubBlocked ? 'blocked' : 'error'
} else if (!executionConfirmed) {
  resultStatus = 'unconfirmed'
} else {
  resultStatus = 'allowed'  // ← Siempre terminaba aquí si allowed=true
}
```

Esta lógica ignoraba `statusResolution.finalUiStatus`, que ya contenía el estado correcto resuelto por el backend.

## 3. Principio

**statusResolution.finalUiStatus manda sobre decision.allowed**

La jerarquía de prioridad es:
1. `statusResolution.finalUiStatus` (si existe)
2. Lógica legacy (fallback)

Nunca mostrar "PERMITIDO" si statusResolution indica:
- `setup_required`
- `reauthorization_required`
- `failed`
- `pending_confirmation`
- `partial`

## 4. Cambios Implementados

### 4.1 Helper getStatusResolution (Execute.tsx:19-43)

```typescript
/**
 * FIX 124.1: Helper seguro para extraer statusResolution de múltiples ubicaciones
 */
function getStatusResolution(response: unknown): StatusResolution | undefined {
  if (!response || typeof response !== 'object') return undefined
  const r = response as Record<string, unknown>

  // Direct statusResolution
  if (r.statusResolution && typeof r.statusResolution === 'object') {
    return r.statusResolution as StatusResolution
  }

  // Inside data
  if (r.data && typeof r.data === 'object') {
    const data = r.data as Record<string, unknown>
    if (data.statusResolution && typeof data.statusResolution === 'object') {
      return data.statusResolution as StatusResolution
    }
  }

  // Inside meta
  if (r.meta && typeof r.meta === 'object') {
    const meta = r.meta as Record<string, unknown>
    if (meta.statusResolution && typeof meta.statusResolution === 'object') {
      return meta.statusResolution as StatusResolution
    }
  }

  return undefined
}
```

### 4.2 resultStatus prioriza statusResolution (Execute.tsx:283-301)

```typescript
// FIX 077 + FEATURE 090 + FIX 124.1: Determinar status correcto
// FIX 124.1: statusResolution.finalUiStatus tiene prioridad absoluta
let resultStatus: ResultStatus
if (statusResolution?.finalUiStatus) {
  // FIX 124.1: Backend ya resolvió el estado correcto
  resultStatus = statusResolution.finalUiStatus
} else {
  // Fallback a lógica legacy solo si no hay statusResolution
  const isMissingCapability = response.error === 'Missing capability' || !!toolProposalId
  if (isMissingCapability) {
    resultStatus = 'missing_capability'
  } else if (!allowed) {
    const hubBlocked = debugSnapshot?.hubAllowed === false || ...
    resultStatus = hubBlocked ? 'blocked' : 'error'
  } else if (!executionConfirmed) {
    resultStatus = 'unconfirmed'
  } else {
    resultStatus = 'allowed'
  }
}
```

## 6. Flujo Corregido

```
1. Backend devuelve respuesta con statusResolution
2. Execute.tsx extrae con getStatusResolution()
3. resultStatus = statusResolution.finalUiStatus (si existe)
4. SecurityResultPanel recibe:
   - status = resultStatus
   - statusResolution = statusResolution
5. SecurityResultPanel.effectiveStatus = statusResolution?.finalUiStatus || status
6. UI muestra estado correcto basado en finalUiStatus
```
